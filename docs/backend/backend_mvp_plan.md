# バックエンドMVP構成整理

## 1. 目的
本プロジェクトでは、ユーザーが「自分自身のキャラクター」と会話できる Flutter アプリを開発する。  
会話手段は以下の2つを対象とする。

- 初回のキャラクター生成
- テキストチャット
- 音声チャット（push-to-talk 方式）

今回の目的は、ハッカソン向けの **動くMVP** を短期間で実装することである。  
そのため、複雑な状態管理や過剰な設計は避け、Google 技術を優先して採用する。

---

## 2. 今回の方針

### 2.1 採用方針
以下の構成をベースとする。

- フロントエンド: Flutter
- 認証: Firebase Authentication（匿名認証）
- データ保存: Cloud Firestore
- バックエンドAPI: Cloud Run
- 生成AI: Vertex AI Gemini

MVP の必須体験は以下とする。

- ユーザー初回入力からキャラクターを生成できる
- 生成されたキャラクターとテキスト会話できる
- 会話履歴を継続利用できる

### 2.2 採用しない方針
今回は以下を対象外とする。

- リアルタイム双方向音声会話
- 画像入力 / 画像生成
- 複雑なキャラクターステータス管理
- フレンド機能やSNS的要素
- 通知、課金、分析基盤
- 過剰なレイヤ分割

---

## 3. プロダクトの考え方
本アプリでは、ユーザーの過去の会話履歴を文脈として活用し、AI がその都度自然な返答を生成する。  
つまり、従来の「固定ステータスを加算するゲーム的な設計」ではなく、**会話履歴ベースで関係性を継続するAIパートナー** として設計する。

また、会話開始前に「ユーザー情報をもとに自分の分身キャラクターが生まれる」体験を MVP の入口として含める。

### 3.0 キャラクター生成
初回起動時に、ユーザー入力をもとにキャラクターの初期設定を作る。

1. Flutter でプロフィール入力
2. バックエンドでキャラクター生成リクエストを受ける
3. 初期プロフィールと相棒設定を保存する
4. 必要なら初期プロンプト用の要約を生成する
5. フロントへ生成結果を返す

### 3.1 テキスト会話
ユーザーがテキストを入力すると、バックエンドが以下を行う。

1. ユーザーの会話履歴を取得
2. 必要に応じて短い要約メモを取得
3. Gemini に文脈付きで問い合わせ
4. 返答を生成
5. Firestore に保存
6. フロントへ返却

### 3.2 音声会話
音声会話は push-to-talk 方式で実装する。

1. Flutter で録音
2. バックエンドへ送信
3. 音声を文字起こし
4. Gemini で返答生成
5. 返答文を音声化
6. Firestore に保存
7. テキストと音声をフロントへ返却

---

## 4. 全体構成

```text
Flutter
 ├─ Firebase Auth（匿名認証）
 ├─ Firestore（ユーザー・キャラクター・会話履歴）
 └─ Cloud Run API
      ├─ キャラクター生成API
      ├─ テキスト会話API
      ├─ 音声会話API
      └─ Vertex AI Gemini 連携
```

### 補足
- MVP の最小構成としては `テキスト会話API` のみでも成立する。
- `音声会話API` は STT / TTS の採用先が決まってから追加する方が安全。

---

## 5. フィードバック

### 5.1 良い点
- Google 系で寄せていて、ハッカソン向けに判断が速い。
- 匿名認証 + Firestore + Cloud Run + Gemini という流れは、MVP として十分現実的。
- 「固定ステータス型ではなく、会話履歴ベースの関係性」を先に定義しているのは良い。

### 5.2 気になる点

#### 音声会話は MVP のコアにすると少し重い
今の文面だと、テキストと音声が同じ重みで MVP 対象に見える。  
ただ、音声会話は以下が追加で必要になる。

- 音声の文字起こし
- 返答の音声合成
- 音声データの送受信

つまり、Gemini だけでは閉じず、STT / TTS の採用先も別途決める必要がある。  
そのため、MVP のコアはまずテキスト会話に寄せた方が安全。

#### 「画像入力を対象外」と現行 UI の整合が少し曖昧
README と現行 UI では `image` 画面が存在する。  
なので docs 上では、

- UI は存在する
- ただし今回の backend MVP では image 連携は対象外

と明記した方が誤解が少ない。

#### diary の扱いがまだ未定義
この docs では chat は具体的だが、diary を今回どう扱うかが書かれていない。  
MVP としては、

- 今回は diary backend はやらない
- もしくは chat の要約メモだけ作る

のどちらかを決めた方が実装順が明確になる。

#### Cloud Run を選ぶ理由を一言足した方がよい
Cloud Run 自体は妥当。  
ただし docs に「なぜ Functions ではなく Cloud Run か」がないので、以下を補足すると筋が通る。

- Flutter から直接 AI を叩かせたくない
- HTTP API として明確に切りたい
- 後でログ・認証・プロンプト制御を集約したい

## 6. 最低限決めておくべきデータ

少なくとも以下の保存単位は先に固定した方がよい。

### users

```text
users/{userId}
- createdAt
- updatedAt
- displayName
- goal
- partnerStyle
- weakPoints: string[]
```

### characters

```text
characters/{characterId}
- userId
- name
- personaPrompt
- avatarState
- createdAt
- updatedAt
```

### chatThreads

```text
chatThreads/{threadId}
- userId
- createdAt
- updatedAt
- lastMessageAt
```

### chatMessages

```text
chatThreads/{threadId}/messages/{messageId}
- role: user | assistant
- text
- inputType: text | voice
- createdAt
```

## 7. 最初に作る API

MVP の最初の一歩としては、以下の 2 本に絞るのが良い。

### `createAnonymousSession`
- 匿名認証ユーザーに対応する初期プロフィールを作る

### `createCharacter`
- 初回入力を受けてプロフィールとキャラクター初期設定を作る
- user と character の紐付けを保存する
- 会話開始に必要な初期状態を返す

### `sendChatMessage`
- ユーザー発話を保存する
- 会話履歴を取得する
- Gemini に問い合わせる
- assistant 応答を保存する
- フロントへ返す

音声会話は、上記のキャラクター生成とテキスト会話が安定してから追加する方が安全。

---

## 8. 修正したおすすめ方針

現時点では以下の整理が最も進めやすい。

1. MVP の必須は「匿名認証 + キャラクター生成 + テキスト会話」
2. 音声会話は stretch goal
3. image backend は今回の対象外なら明記
4. diary backend は後段に回すか、要約メモだけに絞る

つまり、方向性はかなり良いですが、  
**MVP の中心はまず「キャラクター生成 + テキスト会話」を成立させること** に寄せた方が、実装も説明もぶれにくいです。
