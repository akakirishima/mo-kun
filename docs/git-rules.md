# Git Rules

## 目的

ハッカソン期間中の並行開発で、`main` を壊したり、レビューしづらい大きすぎる差分になるのを避けるための最低限のルールです。

とはいえ、今はまだプロトタイプ段階なので `main` のプロテクトは入れておらず、直接 push もできる状態です笑 

なので、開発期間中のガチリポジトリでは、みんなで意識して守っていく、くらいの運用を想定しています！

## ブランチルール

- `main` には直接 push しない
- 作業は必ず `main` から新しいブランチを切って始める
- ブランチ名は、何の作業か分かる名前にする

例:
- `feat/login-ui`
- `fix/home-scroll`
- `docs/setup-guide`
- `chore/flutter-bootstrap`

## コミットルール

- コミットはできるだけ 1 つの目的に絞る
- コミットメッセージは短く、見て内容が分かる形にする
- Issue がある作業は、コミットメッセージの末尾で Issue 番号を紐付ける
- 形式は基本的に以下を使う
  - `feat: ...`
  - `fix: ...`
  - `docs: ...`
  - `chore: ...`
  - `refactor: ...`
  - `test: ...`

例:
- `docs: Git運用ルールを追加 (#1)`
- `feat: 初回オンボーディング画面を追加`
- `fix: ホーム画面のスクロール崩れを修正`

## PR ルール

- 変更は PR ベースで `main` に取り込む
- 1 つの PR では 1 テーマに絞る
- PR には最低限、以下を書く
  - 概要
  - 何ができるようになったか
  - 何を確認したか
- UI 変更がある場合は、スクリーンショットを付ける

## マージ前にやること

- `flutter analyze` を通す
- `flutter test` を通す
- 少なくとも自分が変更した範囲は手元で動作確認する

## 補足

- 大きめの作業は、できれば先に Issue を作ってから着手する
- 迷ったら「大きい 1 本」より「小さい PR に分ける」を優先する
- 他の人の未反映変更を上書きしないように気をつける
